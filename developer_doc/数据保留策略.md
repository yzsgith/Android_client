# 数据保留策略

> 本文档定义应用升级时如何保留用户数据（数据库、配置、文件、任务等），以及必要的迁移方案。  
> **模块说明**：与 Android 框架强相关的代码（Room、SharedPreferences、WorkManager、Application）保留在 `:app` 模块；纯逻辑数据类（如 `SyncStatus`）已迁移至 `:domain` 模块。

## 一、核心原则

- **默认保留**：应用升级时，Android 系统会自动保留应用的私有数据（`/data/data/包名/` 下的文件、数据库、SharedPreferences）。
- **结构变化需迁移**：若数据库表结构、SharedPreferences 键名、文件格式发生变化，必须提供迁移逻辑，否则可能导致崩溃或数据丢失。
- **任务持久化**：WorkManager 的任务队列会保留，但任务类名或参数变化时需要手动清理旧任务。

## 三、各模块数据保留方案

### 1. Room 数据库（`:app` 模块）

**文件位置**：`data/local/AppDatabase.kt`

- **策略**：每次版本升级时，若修改了实体（Entity）结构，必须增加数据库版本号并提供 `Migration` 对象。
- **示例代码（含 TODO 注释）**：

```kotlin
// AppDatabase.kt
@Database(
    entities = [RawDataEntity::class, TransformedDataEntity::class, 
                ConsumableDataEntity::class, SyncQueueEntity::class],
    version = 1,  // TODO: 每次结构变更时增加版本号
    exportSchema = true
)
abstract class AppDatabase : RoomDatabase() {
    // ... DAO 抽象方法

    companion object {
        // TODO: 根据版本变化添加 Migration 对象，例如从版本1到2
        val MIGRATION_1_2 = object : Migration(1, 2) {
            override fun migrate(database: SupportSQLiteDatabase) {
                // 示例：新增字段
                database.execSQL("ALTER TABLE raw_data ADD COLUMN source TEXT DEFAULT 'sensor'")
                // TODO: 根据实际需求编写迁移SQL
            }
        }

        fun getInstance(context: Context): AppDatabase {
            return Room.databaseBuilder(context, AppDatabase::class.java, "app_database")
                .addMigrations(MIGRATION_1_2)  // TODO: 添加所有迁移对象
                .fallbackToDestructiveMigration()  // 仅开发期使用，生产环境请移除
                .build()
        }
    }
}
