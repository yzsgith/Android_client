# 目标设计与优化建议

## 第一部分：项目设计目标

本项目的核心目标是构建一个**支持用户主动交互与后台自动采集的智能数据终端**，实现传感器数据的本地存储、处理、可靠上传至 Netlify 后端，并具备离线支持和动态配置能力。具体包括：

1. **双模式交互**
    - 用户主动模式：通过 WebView 加载服务器前端页面，用户可完成文件上传、下载、查询、修改等操作。
    - 后台自动模式：无需用户参与，自动采集传感器（如心率、加速度）数据，并进行本地处理与上传。

2. **数据处理流水线**
    - 抽象原始数据（RawData）→ 转化数据（TransformedData）→ 消费数据（ConsumableData）三个阶段，支持多种传感器类型的灵活扩展。
    - 提供 `DataProcessor`、`ProcessorChain` 等接口，实现可组合的数据转换逻辑。

3. **本地优先与离线同步**
    - 数据优先存入 Room 本地数据库，网络恢复后通过 WorkManager 自动上传，确保数据不丢失。
    - 支持分批上传、自动重试，适应不稳定的网络环境。

4. **配置持久化与动态化**
    - 使用 DataStore 保存用户设置（如服务器地址、上传间隔、传感器开关），应用升级后配置保留。
    - 支持运行时通过设置界面修改配置，无需重新编译 APK。

5. **模块化分离**
    - 将纯逻辑代码（数据结构、处理器接口）抽取为独立的 `domain` 模块，不依赖 Android 框架，提高可测试性和复用性。
    - `app` 模块负责 Android 相关实现（UI、传感器采集、WorkManager 调度等）。

6. **文档驱动开发**
    - 通过 `总览.md`、`全部核心任务.md`、`任务执行顺序.md`、时序图等文档管理项目，确保代码与设计一致，便于二次开发。
    - 每个文件以 `// TODO: 作用 --` 注释说明职责，并建立双向链接。

## 第二部分：优化建议

基于当前项目进展，从架构、功能、流程、运维四个方面提出优化建议：

### 一、架构与代码优化

1. **增强单元测试**
    - 为 `domain` 模块中的处理器编写单元测试（使用 JUnit + `kotlinx-coroutines-test`），验证数据转换逻辑。
    - 使用 `MockWebServer` 测试 `Uploader` 的重试和超时行为。
    - 测试 `SensorDataRepository` 和 `AppPreferences` 的关键方法。

2. **引入依赖注入（可选）**
    - 当前通过 `Application` 单例提供依赖，对于中型项目足够。若未来模块增多，可考虑 Hilt 简化管理，特别是 Worker 的依赖注入（目前通过 `ServiceLocator` 模式，可接受）。

3. **细化错误处理与日志**
    - 在 `SensorUploadWorker` 中区分可重试错误（网络异常、5xx）和不可重试错误（数据格式错误），避免无限重试。
    - 将关键操作日志写入本地文件（如使用 `Timber` + `FileTree`），便于远程排查。

4. **性能调优**
    - 调整 `BATCH_SIZE`（当前 50），根据网络状况和数据大小优化。
    - 为 `raw_data` 表的 `synced` 和 `timestamp` 字段创建复合索引，加速分批查询。

### 二、功能扩展建议

1. **支持更多传感器类型**
    - 扩展 `SensorData.kt`，添加陀螺仪、磁力计、气压计等模型，并实现对应处理器（如 `GyroscopeProcessor`）。

2. **数据可视化与实时监控**
    - 在 WebView 页面中增加图表库（如 ECharts），通过 JS Bridge 获取本地数据，展示心率曲线、步数统计等。

3. **数据清理策略**
    - 实现定期清理任务（WorkManager），删除已同步超过 30 天的旧数据，避免数据库膨胀。

4. **多账户支持**
    - 扩展 DataStore 配置为按用户隔离（如 `user_settings`），或上传时附带 `userId`，为多用户场景预留。

### 三、开发流程与文档

1. **维护任务文档**
    - 持续更新 `全部核心任务.md` 和 `任务执行顺序.md`，确保任务状态与代码同步。
    - 每次新增功能或修改架构时，更新 `总览.md` 和 `时序生命线_作用.md`。

2. **编写开发者快速上手指南**
    - 创建 `developer_doc/开发者快速上手.md`，包含环境配置、常用 Gradle 命令、模块结构说明、如何运行和调试。

3. **代码审查与规范**
    - 强制要求 PR 附带文档更新，使用 `ktlint` 或 `detekt` 进行静态检查，保持代码风格一致。

### 四、部署与运维

1. **持续集成（CI）**
    - 配置 GitHub Actions 或 GitLab CI，自动运行单元测试、静态检查，并生成测试报告。

2. **构建变体配置**
    - 利用 `buildTypes`（debug/release）和 `productFlavors`（dev/prod），自动切换 API 端点、日志级别等。

3. **监控与崩溃上报**
    - 集成 Firebase Crashlytics，监控应用运行时错误，尤其是 Worker 的失败率。

### 五、下一步具体行动（优先级排序）

1. **完成 T-014**：在 `WebViewBridge` 中实现 `startAutoUpload` 和 `stopAutoUpload`，通过 WorkManager 动态启停 `SensorUploadWorker`。
2. **集成真实后端**：将测试端点替换为 Netlify 函数，验证完整上传流程。
3. **完善设置界面**：确保 `SettingsActivity` 修改配置后，`SensorUploadWorker` 能使用新 URL（需要重建 WorkManager 任务或动态更新 `baseUploadRequest`）。
4. **编写单元测试**：为 `Uploader`、`AppPreferences`、`SensorDataRepository` 等核心类编写测试。
